# Distributed Spot Controller — Implementation Q&A Answers

This document collects the chosen implementation answers for the Distributed Spot Controller, based on the implementation questionnaire.

---

## 1. Auth model for dashboard
**Question:** JWT bearer vs cookie session?  
**Chosen answer:** C — Hybrid: cookie session now, architecture ready for JWT later  
**Explanation:** For v1, the dashboard will use server-side cookie sessions because it is simpler, safer, and better suited to a browser-only admin panel. The system should still be structured so JWT-based API auth can be added later for external or non-browser clients without redesigning the whole backend.

## 2. Node download auth
**Question:** How should `/api/spots/{id}/download` authenticate node requests?  
**Chosen answer:** B — Signed short-lived per-file download URLs  
**Explanation:** Spot downloads will use temporary signed URLs generated by the server during sync. Each URL should be scoped to the target node, the specific spot, and a short expiration window. This reduces the impact of URL leakage and avoids reusing the long-lived node token for every file transfer.

## 3. Node provisioning flow
**Question:** How are node tokens created, rotated, and revoked?  
**Chosen answer:** B — One-time pairing / enrollment flow now  
**Explanation:** New nodes will start untrusted and go through an explicit enrollment flow, such as a pairing code or admin approval process. After approval, the server issues a permanent node-specific token stored by the app. Rotation remains an admin-controlled action, and revocation is handled primarily by disabling the node server-side, with token replacement or invalidation when needed.

## 4. Assignment model
**Question:** Global all-spots vs per-node vs per-group?  
**Chosen answer:** A — Global all-spots to every node  
**Explanation:** The primary goal of the system is synchronized reproduction of the same spot across multiple nodes. Keeping the full active spot catalog on every node simplifies synchronization logic, reduces the chance that a scheduled or manual synchronized playback fails because one node is missing a file, and makes fleet-wide playback orchestration much more predictable. The tradeoff is higher storage and bandwidth use, but for this project that is acceptable because synchronized availability is more important than assignment granularity.

## 5. Scheduler overlap policy
**Question:** If next interval fires while a node is still playing, what happens?  
**Chosen answer:** A — Hard cut current playback and start the next scheduled spot  
**Explanation:** The system should use the same `PLAY` command semantics for both manual interaction and autoplay. When a new scheduled play is triggered, it should replace any current playback immediately. This keeps the command model simple, avoids separate autoplay-only behavior, and preserves a single authoritative playback action across the system.

## 6. Manual play vs scheduler arbitration
**Question:** If autoplay is running and an admin triggers manual play, who wins?  
**Chosen answer:** C — Manual play wins, autoplay timer resets from the manual play  
**Explanation:** Manual play is treated as an immediate authoritative `PLAY` command. It replaces current playback and becomes the new timing anchor for autoplay. Autoplay remains enabled, but the next scheduler trigger is recalculated from the time of the manual play so the operator does not get an unexpected automatic override a few seconds later.

## 7. `SET_ENABLED=false` during playback
**Question:** Stop immediately, or block only future plays?  
**Chosen answer:** C — Disable blocks future plays; use STOP separately for immediate silence, with a combined operator action “Stop and disable”  
**Explanation:** The system should keep command semantics explicit. `SET_ENABLED=false` should prevent future participation in playback, while `STOP` should control current audio. For operator convenience, the dashboard should also provide a combined action that sends both commands in sequence when the goal is to silence a node immediately and exclude it from future playback.

## 8. Protocol compatibility/versioning behavior
**Question:** Reject unknown versions, support range, or another policy?  
**Chosen answer:** C — Exact major-version compatibility, tolerate backward-compatible minor additions  
**Explanation:** The protocol should evolve in a disciplined way. Nodes and server must share the same major protocol version for compatibility, while backward-compatible minor additions inside the same major line can be tolerated. Receivers should ignore unknown extra fields but reject missing required fields and unsupported major versions with explicit errors.

## 9. Idempotency on write endpoints
**Question:** Do we need idempotency keys or defined duplicate semantics?  
**Chosen answer:** D — State-setting endpoints idempotent by design, command endpoints documented separately  
**Explanation:** The system should distinguish clearly between state-setting operations and command operations. Endpoints that set resource state, such as enable flags or scheduler configuration, should be idempotent by design. Command endpoints such as `PLAY` and `STOP` should keep explicit execution semantics documented separately, instead of introducing a full generic idempotency-key layer in v1.

## 10. Error schema standardization
**Question:** How should REST and WS errors be modeled?  
**Chosen answer:** D — Human-readable messages only  
**Explanation:** The chosen approach for v1 is to keep error responses simple and focused on operator readability rather than a strongly standardized machine-oriented schema. Errors should still be clear, concise, and consistent in wording, but they will primarily be designed to help a human operator understand what went wrong quickly rather than enforce a rich structured contract.

## 11. Bulk command partial success response
**Question:** How should `/play` and `/stop` report per-node results?  
**Chosen answer:** D — Wait for node acknowledgements before returning  
**Explanation:** For bulk playback and stop commands, the API should wait for node acknowledgements before responding so the operator receives a stronger immediate confirmation of which nodes actually accepted the command. This favors clearer operator feedback at command time over the simpler fire-and-return approach, even though final playback completion still remains an asynchronous concern.

## 12. Optimistic concurrency for scheduler config
**Question:** How should admin update races be handled?  
**Chosen answer:** E — Optimistic concurrency for config, direct command semantics for start/stop  
**Explanation:** Scheduler configuration should be treated as shared mutable state and protected with a version or revision check so stale admin updates cannot silently overwrite newer ones. Start and stop operations should remain command-style runtime actions with explicit current-state behavior, separate from config editing semantics.

## 13. Upload validation policy
**Question:** What upload validation rules should be enforced?  
**Chosen answer:** B — Extension + MIME validation only  
**Explanation:** For v1, upload validation will remain lightweight and operator-friendly. The backend will accept only approved audio formats based on file extension and declared MIME type, rejecting obviously unsupported content while avoiding a heavier media probing or transcoding pipeline in the first implementation.

## 14. Single scheduler instance guarantee
**Question:** How do we avoid double scheduler firing?  
**Chosen answer:** A — Trust single backend instance only  
**Explanation:** For v1, the deployment model assumes a single backend process as the only scheduler authority. This keeps the architecture and operations simple in the first implementation. The team must enforce deployment discipline so only one backend instance is active at a time while autoplay is enabled.

## 15. WS connection manager durability
**Question:** Is in-memory enough, and what is the migration path?  
**Chosen answer:** B — In-memory live connections + DB-persisted last-known node state  
**Explanation:** For v1, active WebSocket objects will live only in backend memory, which is acceptable because the system runs on a single backend instance. To preserve operator visibility and improve recovery, the backend will also persist each node’s last-known state, last seen time, and operational status in the database. After a backend restart, nodes reconnect and rebuild the live connection registry.

## 16. Heartbeat timeout strategy
**Question:** What timeout constants and rules should be used?  
**Chosen answer:** B — Fixed heartbeat + stale/degraded state + offline timeout  
**Explanation:** The system should use a simple but nuanced liveness model. Nodes will send heartbeats at a fixed interval, and the backend will first mark a node as stale/degraded before declaring it offline. This gives operators earlier warning of degraded connectivity without causing overly aggressive offline transitions.

## 17. Crash recovery semantics
**Question:** How should stale playback state be recovered after restart?  
**Chosen answer:** A — Reset live state on restart and rebuild from reconnects  
**Explanation:** After a backend restart, in-memory live playback assumptions should be discarded. The database may retain historical last-known node information, but it should not be treated as authoritative current playback truth. Nodes must reconnect and report fresh status so the backend can rebuild an accurate live view safely.

## 18. Command delivery guarantees
**Question:** At-most-once or at-least-once?  
**Chosen answer:** D — At-most-once delivery with explicit operator retry on failure  
**Explanation:** The system should prefer avoiding accidental duplicate command execution, especially for `PLAY`, which has immediate replace semantics. Commands are sent once over the active connection and awaited for acknowledgement where applicable. If acknowledgement does not arrive, the failure is surfaced clearly to the operator instead of being retried blindly in the background.

## 19. Storage growth controls
**Question:** What retention and cleanup policies should be used?  
**Chosen answer:** C — Basic automatic cleanup with defined retention rules  
**Explanation:** The system should apply simple automatic cleanup policies so storage does not grow indefinitely on either the server or the nodes. Active spots remain available, while removed or inactive content is cleaned up according to clear retention rules. Nodes delete files that are no longer part of the active catalog during sync, temporary files are cleaned immediately, and logs/history are kept for a defined time window.

## 20. Plain token in `HELLO`
**Question:** What security posture is acceptable for v1?  
**Chosen answer:** B — Plain token in `HELLO`, but only over WSS/TLS  
**Explanation:** Node authentication will use a plain token in the `HELLO` message, but only through encrypted WSS/TLS transport. Since nodes will not all be on the same LAN, unencrypted WebSocket transport is not acceptable. TLS becomes mandatory so node credentials are protected in transit while keeping the protocol and implementation relatively simple for v1.

## 21. Rate limits / lockout policy on admin login
**Question:** What protections should exist?  
**Chosen answer:** C — Per-IP + per-username limiting with temporary lockout  
**Explanation:** Admin login should be protected both against repeated attempts from the same source and against targeted attacks on a specific account. The system will apply rate limits by IP and by username, and temporary lockouts will be triggered after repeated failed attempts. This gives a strong and understandable protection baseline without adding heavier mechanisms like CAPTCHA in v1.

## 22. Audit trail for admin actions
**Question:** What should be logged about admin changes?  
**Chosen answer:** A — Minimal audit only for sensitive actions  
**Explanation:** For v1, the audit trail will stay focused on the most important security and operational events rather than every possible admin interaction. This keeps implementation and storage simpler while still preserving visibility into the actions that matter most, such as authentication events, node enable/disable actions, provisioning approvals, scheduler start/stop, and playback control operations.

## 23. File download authorization leakage risk
**Question:** How do we prevent unauthorized spot download?  
**Chosen answer:** B — Signed URL bound to node identity and spot identity  
**Explanation:** Download URLs should be temporary and scoped, not generic. Each signed URL will be bound to the specific node and the specific spot it is meant to fetch, with an expiration time enforced by the backend. This limits the impact of leakage and prevents one node’s download link from being reused more broadly.

## 24. Service supervision and restart policy
**Question:** How should the host keep the app running?  
**Chosen answer:** B — systemd service on the host  
**Explanation:** For v1, the backend should run as a single Linux service supervised directly by the host operating system. `systemd` will provide automatic startup on boot, restart on failure, and standard service management behavior without adding the extra operational layer of containers in the first deployment.

## 25. Health/readiness endpoints
**Question:** What should health checks include?  
**Chosen answer:** A — One simple `/health` only  
**Explanation:** For v1, the backend will expose a single simple health endpoint that reports whether the process is alive. This keeps operations and implementation straightforward in the first deployment, without introducing separate readiness semantics or more detailed diagnostic surfaces.

## 26. Log rotation, structured logs, and alerting
**Question:** What logging/alerting baseline is needed?  
**Chosen answer:** C — Plain text logs with host-managed rotation, no alerting yet  
**Explanation:** For v1, logs should stay easy for a human operator to read directly, while retention and rotation are handled by the host environment. This keeps the operational model simple and aligned with a `systemd`-managed deployment, without adding a separate alerting or structured telemetry layer in the first version.

## 27. Backup plan for PostgreSQL + spot storage
**Question:** What backup strategy should be used?  
**Chosen answer:** A — Manual backups only  
**Explanation:** For v1, backups will be handled manually by the operator rather than through an automated backup pipeline. This keeps the first deployment operationally simple, even though it relies on operator discipline and is less resilient than an automated backup strategy.
